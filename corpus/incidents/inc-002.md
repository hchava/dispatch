# INC-002 — orders-fact row count dropped 94%

## What happened
The orders-raw upstream pipeline deployed a schema change at 02:14 UTC: added a new nullable column `delivery_estimate_ts TIMESTAMP NULL`. The ingestion job for orders-fact uses partition pruning conditioned on `WHERE updated_at IS NOT NULL AND order_status IS NOT NULL`. The new column caused a silent join condition change that resulted in 94% of rows being filtered out.

## Detection
Row count quality rule fired at 08:43 UTC (6.5 hours after root cause — the SLA check runs at job completion).

## Why it took so long to detect
The ingestion job completed without error. The quality rule only fires post-completion. No schema validation at ingestion boundary. No upstream change notification.

## Resolution
1. Traced row count drop to partition pruning via EXPLAIN output
2. Identified the new nullable column in the upstream schema diff
3. Fixed ingestion query to handle the new column explicitly
4. Triggered backfill for the affected partition (2026-04-22)
5. Verified row count recovered to within 0.2% of 7-day average

## Follow-up
- Added schema validation step at ingestion boundary (fail fast on unexpected columns)
- Required upstream owners to update the data contract before deploying schema changes
- Added schema diff check to the CI pipeline for orders-raw
